# 视频总结：那些 Agent 神文没告诉你的事

> **来源**：[数字黑魔法 (@nft8888)](https://www.youtube.com/@nft8888)
> **链接**：https://youtu.be/b_9D7T0n4RA
> **发布日期**：2025-12-21
> **类型**：AI Agent 搭建实操经验分享

---

## 一句话概括

**别照着 Anthropic 的架构图抄作业——AI Agent 必须被问题「逼着长大」，而不是一开始就设计得很优雅。**

---

## 背景

作者（数字黑魔法频道主）在开发一个视频 AI Agent 的过程中，看到了 Anthropic 等大厂发布的 Agent 架构设计文章，觉得写得太好了，决定「抄」。于是做了一轮大升级：系统提示词升级、上下文隔离、加 memory、搞了一套自认为很优雅的架构。

结果？**系统变得更烂了。** Token 花费更高，效果没变好，失败率反而上升，有些 bug 根本没法 debug。

后来他想明白了一件事：**网上的文章只展示毕业设计的完整图纸，没人告诉你系统是怎么一步步长大的、中间踩了多少坑。**

---

## 核心观点：AI Agent ≠ 传统软件

传统软件开发里，先规划好 Docker 部署、微服务拆分，最多就是前期慢一点，不会出大事。

但 AI Agent 不一样——**它本身就是一个非确定性系统**。你在不确定性上面再搭一层精致架构，等于是不确定性叠加不确定性。

---

## 演进路线图（7 个阶段）

### 第 1 阶段：能用 API Call 解决的，别上 Agent

作者自曝蠢事：只是想把一段话总结成一句话，结果直接上了 Plan and Execute 模式——先让 AI 规划，再让它执行。原本一个 API Call 搞定的事，链路先复杂了，效果反而差了。

**法则**：一个 API Call 能解决的问题，就不要用 Agent。

### 第 2 阶段：多步骤流程 → Workflow 链式结构

当任务确实需要多个步骤时（比如先搜索、再分析、再输出），不代表就需要 Agent。一个固定顺序的 Workflow（链式调用）就够了。

**关键区分**：步骤多 ≠ 需要 Agent。如果每一步的输入输出是确定的，用 Workflow 串起来就行。

### 第 3 阶段：引入对话式 Agent（真正需要交互时）

作者曾开发「一键生成特效」功能，以为用户点个按钮就能出结果。现实是：**很难一次生成到满意**——风格不对、节奏不对、只想改某个小细节。

这类任务不是对错题，是审美题，需要人反复指导和修改。如果继续用按钮模式，前端会变成一堆控制面板。**这时候才真正需要 Agent。**

**何时引入 Agent**：
- 任务需要人参与反馈（不是一次性的）
- 功能选项太多，按钮已经放不下了
- 任务本质上是「审美题」或「开放题」

### 第 4 阶段：技术选型——先跑起来，别被框架迷了眼

确定要做 Agent 后，作者立刻犯了一个典型错误：想选最强最完整的框架（LangGraph 之类）。觉得问题复杂，链很长，必须上最硬核的后端框架。

**错在哪**：他混淆了两种「长链」的概念——链长不等于必须用复杂调度。

最终他选了看起来平平无奇的 **AI SDK**（集成度高、上手快），先把东西跑起来。

**核心教训**：
- 复杂框架会诱导你瞎设计——还没跑任何东西就开始画节点、设计数据流
- 你连最简单的问题能不能解决都不知道，这就是闭门造车
- 即使选了复杂框架，也要先用最简单的方式跑一遍，**跑出 baseline**

### 第 5 阶段：加工具——能力缺失时别改提示词

跑起来后很快遇到现实：做不好不是提示词不够好，而是**能力缺失**。比如让 AI 做动效设计，期待它参考网上流行设计，但它根本拿不到这些数据。

这时候再优化提示词没用，**正确动作是加工具**：
- 想让它参考网上信息 → 加搜索工具
- 想让它写的代码能跑 → 加代码验证工具

加了三四个工具后，系统开始像一个真正的 Agent——它会自己想用哪个工具、串联使用，出现了**「涌现」效果**（1+1 > 2）。

**但很快进入诡异阶段**：工具越加越多，Agent 性能反而持续变差。不是模型不行，是**上下文被污染了**——

两个本质不同的任务被混在一个 Agent 里：
- **设计任务**：关心用户意图、视觉风格，需要发散性信息
- **代码任务**：关心接口结构、输出格式，需要精确信息

两者混在一起，互相干扰。

### 第 6 阶段：Sub-Agent + 上下文隔离 + 记忆系统

上下文互相污染后，才真正需要引入 **Sub-Agent 架构**：
- **顶层规划者（Planner）**：知道全局信息，负责调度
- **执行者（Executor）**：各自只看到自己需要的信息

**关键原则**：如果 sub-agent 和顶层看到的上下文一样，那这个 sub-agent 就没有意义。

但拆分后立刻遇到新问题——**长内容传递**：
- 用户提交了一段代码让 AI 修改，规划者看到了代码，但怎么把它原封不动传给代码执行者？
- 让规划者输出一遍？**两个糟糕点**：
  1. 为 copy-paste 付费（大量 output token）
  2. 模型不擅长机械复制，可能偷偷「修」了你的 bug

**解决方案**：引入**记忆系统**——把代码存进去，规划者只传一个引用 ID，执行者直接从记忆中读取。

效果：token 成本直降，错误率明显降低。

### 第 7 阶段：存过程、做评估

到这一步系统已经很复杂了：Sub-Agent、上下文隔离、记忆系统。怎么 debug？

**答案：把每一次运行的全过程都存下来。**

存的不是结果，是**过程**：
- 用了哪些工具
- 工具调用顺序
- 每步消耗的 token
- 哪些上下文没被用到

回头看完整流程，才能知道怎么优化。

---

## 最核心的反思

> 那些成型的文章更像是**毕业设计的完整图纸**——当你做过所有实验、踩过坑后，回头看会发现设计完美精巧。但如果第一天就照着毕业设计施工，大概率连第一根梁都会弄歪。

**问题从来不是架构不够优雅，而是你还没有走到需要那个架构的阶段。**

---

## 演进路线总结

```
API Call（一句话搞定）
  ↓ 任务变复杂
Workflow（固定链式调用）
  ↓ 需要人参与/选项太多
对话式 Agent（简单 SDK 先跑起来）
  ↓ 能力不够
加工具（搜索、代码验证等）
  ↓ 上下文互相污染
Sub-Agent + 上下文隔离
  ↓ 长内容传递出问题
记忆系统（外部存储 + 引用ID）
  ↓ 系统复杂到难以调试
存运行过程 + 评估体系
```

每一步都是被问题「逼」出来的，不是提前设计出来的。
